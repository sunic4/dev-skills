# KB - 知识库

## 职责
**raw 接收 → 去重合并 → 质量检查 → 发布正式目录**

KB 是知识流的**唯一整理者**（curator）。其他技能只负责写 raw 和读正式目录。

**核心原则：**
1. **质量 > 数量**：不记一条无用的，也不漏掉一条确实有价值的。
2. **时效性 > 完美性**（新增）：宁可让正式目录有几条质量不完美的条目，也不让高价值的即时知识在 `{project-path}/wiki/kb/raw/` 中等待整理。

## 目录结构

```
{project-path}/wiki/kb/
├── raw/              # 阶段1: 写入区（不可被其他技能检索）
├── patterns/         # 阶段2: 正式发布区
├── lessons/
├── adrs/
├── refs/
└── _archive/         # 归档（过时但保留）
```

## Raw 写入规范（所有技能必须遵守）

各技能写 raw 时使用模板 `templates/raw.md`，确保 cc-kb 整理时输入格式一致。

**各 category 额外必填项**:

| category | 额外必填 |
|----------|---------|
| **pattern** | 代码示例 + 反例 |
| **lesson** | 先错后对对比 + 预防措施 (至少2层5Whys) |
| **decision** | ADR引用或完整决策记录 |
| **reference** | 使用场景说明 |

## 触发方式

| 方式 | 触发时机 |
|------|---------|
| **嵌入式** | 各技能按需写 raw（不调用 cc-kb） |
| **整理发布** | raw ≥3条 / 用户要求 / 检索发现内容在 raw 中 |
| **显式调用** | 用户"记下来" / 复盘批量整理 / onboarding 补充 |
| **时效性快速发布**（新增） | 解决问题后立即写 raw → **直接发布**到正式目录（标注 `{待完善}`），不等整理周期 |

## 跨项目知识复用

### 标记规则

当知识条目具有跨项目通用性时 (如认证模式、错误处理策略、通用数据结构)，标记 `cross_project: true` 并添加 `cross_project_tags`。

**判定标准**:
- 与具体业务逻辑无关 → 可标记
- 可在 2+ 个项目中直接复用 → 可标记
- 仅依赖通用技术栈 (非项目特有) → 可标记

### 跨项目检索

新项目初始化时 (cc-init Phase2)，可扫描已有项目的 `{project-path}/wiki/kb/` 中 `cross_project: true` 的条目：

```
Grep "cross_project: true" in {已有项目}/{project-path}/wiki/kb/**/*.md
→ 按 cross_project_tags 分类
→ 建议用户选择适用的条目复制到新项目
```

### 标签体系 (cross_project_tags)

| 领域 | 示例标签 |
|------|---------|
| 认证授权 | auth, oauth, jwt, rbac |
| 数据处理 | validation, transform, pagination |
| 错误处理 | error-handling, retry, circuit-breaker |
| 测试 | testing, mocking, e2e |
| 部署 | ci-cd, docker, monitoring |
| 通用模式 | caching, logging, config |

> 标签不预设完整列表，按需扩展。新增标签时在本表中补充即可。

## 时效性快速发布通道（新增）

> **核心原则: 今天踩的坑，明天还会踩。宁可先发布不完美的条目，也不让知识在 `{project-path}/wiki/kb/raw/` 中等待。**

**适用条件**（任一满足）:
- 解决了一个 >15min 的问题（lesson）
- 发现了一个有普遍性的 bug 模式
- 用户显式说"记下来"/"这个要记住"

**快速发布规则**:
1. 写入 `{project-path}/wiki/kb/raw/`（如常）
2. **立即**执行去重检查（Step 1）
3. 快速质量评估：只检查 Q1(有标题) + Q3(有"为什么") → 通过即可发布
4. 发布时标注 `status: draft` + `{待完善: 缺少代码示例/适用场景}`
5. 正常整理时再补充缺失项，更新 `status: verified`

| 标准 | 正式发布 | 快速发布 |
|------|---------|---------|
| 质量门槛 | Q1-Q5 全通过 | Q1 + Q3 通过即可 |
| status | verified | **draft** |
| 代码示例 | 必须 | 可缺（标注 `{待完善}`） |
| 适用场景 | 必须 | 可缺 |
| 可被其他技能检索 | ✅ | ✅（标注 `status: draft` 提醒读者） |

## 整理流程

```
raw 条目 → 去重(2步) → 质量门槛(Q1-Q5) → 发布/保留/归档
```

### Step 1: 去重逻辑

**精确匹配优先，语义重叠 AI 辅助判断。**

#### 1.1 精确匹配

```
Grep "{标题关键词}" in {project-path}/wiki/kb/{对应子目录}/*.md
```

| 结果 | 动作 |
|------|------|
| 标题相同 + 同目录 | **更新旧条目**：追加差异段落 + 更新时间戳 |
| 正文 >80% 相同 | 同上，视为变体 |

#### 1.2 语义去重（AI 增强，新增）

> **核心改进**: 利用 AI 的语义理解能力，而非仅依赖 Grep 关键词匹配。

```
对每个 raw 条目：
1. 提取核心语义（问题是什么、解决方案是什么）
2. 与正式目录中已有条目的语义对比
3. 判断是否为同一知识的不同表述
```

| 场景 | 判断方法 | 动作 |
|------|---------|------|
| 语义等价（同一问题同一方案，表述不同） | AI 判断核心问题和解决方案一致 | **合并**：选择更完整的版本，补充差异部分 |
| 语义相关（不同问题，共同根因） | 问题不同但解决思路相似 | 分别保留，互加 `see_also:` 链接 |
| 语义包含（新条目是旧条目的子集） | 旧条目覆盖了新条目的内容 | 不发布新条目，补充旧条目缺失细节 |
| 无法确定 | AI 置信度 < 80% | 保持 `{project-path}/wiki/kb/raw/` + 标注 `{待确认: 与 XXX 语义可能重叠}` |

> 版本演化：新方案完全替代旧 → 旧入 `_archive/` + 标记 `superseded_by`；新旧互补 → 追加 `## 更新` 章节。

### Step 2: 质量门槛（Q1-Q5）

| # | 检查项 | 通过标准 | 不通过处理 |
|---|--------|---------|-----------|
| Q1 | 有明确标题 | 一眼知道讲什么 | 补充标题 |
| Q2 | 有可运行代码示例 | 可直接使用（纯概念标注 `[概念]`） | 补充真实代码 |
| Q3 | 解释了"为什么" | 不只是"怎么做" | 追加原因分析 |
| Q4 | 标注适用场景和局限性 | 知道何时用/不用 | 补充说明 |
| Q5 | 不是纯链接/引用 | ≥3行有效内容 | 追加到目标条目的参考章节 |

**判定**:

| 结果 | 动作 |
|------|------|
| 5/5 + 无冲突 | **自动发布**到正式子目录 |
| 3-4 + 小缺陷 | 自动补全后发布，标注 `{已自动补全}` |
| <3 或需人工判断 | 保持 `{project-path}/wiki/kb/raw/` + 标注 `{待整理}` |

## 输出格式

```markdown
---
title: "{标题}"
type: knowledge
status: verified
category: pattern | lesson | decision | reference
source: "{来源技能} - {日期}"
see_also: []
supersedes: ""
created: "YYYY-MM-DD"
updated: "YYYY-MM-DD"
---

# {标题}

## 一句话总结
{一句话说清楚}

## 问题/场景
{什么情况下遇到}

## 解决方案
{具体做法 + 代码示例}

## 为什么这样做
{原因分析}

## 适用场景
- ✅ {适合}
- ❌ {不适合}

## 局限性
{什么时候不 work}
```

## 类型特殊要求

| 类型 | 特殊规则 |
|------|---------|
| **pattern** | 必须含完整可复制代码 + 适用技术栈版本 + 至少一个反例 |
| **lesson** | 先错后对（对比结构）+ 5 Whys 至少2层 + 预防措施 |
| **decision** | 有 ADR 则只写引用 + 一句话结论；无 ADR 则完整写 |
| **reference** | 允许速查表格式，但至少一行使用场景说明 |

## 归档流程（2步）

**触发条件**: 被新条目替代 / 技术栈 EOL≥1年 / 无引用≥6个月 / 用户要求

1. **标记 + 移动**: frontmatter 加 `status: deprecated` + `archive_reason` + `archived_at` → 移入 `_archive/{YYYY-MM}-{slug}.md`
2. **更新引用**: 替代条目加 `supersedes:` 链接；其他条目的 `see_also:` 指向归档路径

> **永不删除**。归档条目保留在 `_archive/`。

## 维护规则

| 规则 | 说明 |
|------|------|
| 及时记录 | 解决后立即写 raw，不依赖记忆 |
| 定期整理 | raw ≥ 3 条触发整理 |
| 单一事实源 | 同一知识点只在正式目录存一处 |
| 相互链接 | 整理时建立 see_also 关系 |
| **主动推送**（新增） | 各技能在执行时主动检索 KB 正式目录（见下方知识拉取机制） |

## 知识拉取机制（新增）

> **核心原则: 知识不应只在"写入时"产生，更应在"需要时"主动推送。**

**各技能的 KB 检索时机**（取代原有的"按需检索"）:

| 技能 | 主动检索时机 | 检索内容 |
|------|------------|---------|
| cc-feat-design | 设计开始前 | 检索与当前功能相关的 patterns + lessons |
| cc-feat-impl | 编码涉及特定技术时 | 检索该技术的 patterns + references |
| cc-fix-analyze | 根因分析时 | 检索相似问题的 lessons |
| cc-arch | 技术选型时 | 检索相关 adrs + patterns |
| cc-review | 审查发现问题时 | 检索已知反模式的 lessons |

**检索方法**:
```
1. 从当前任务提取关键词（技术名、问题类型、模块名）
2. Grep `{project-path}/wiki/kb/{对应子目录}/*.md` 的标题和关键词
3. 如有时效性快速发布的条目(status: draft)，提示"此经验尚未完善验证，仅供参考"
```

> **KB 读写矩阵**（权威定义）:

| 技能 | 写入 {project-path}/wiki/kb/raw/ | 读取正式目录 |
|------|----------|-------------|
| cc-feat | pattern, lesson | patterns/, lessons/ |
| cc-fix | lesson | lessons/ |
| cc-arch | decision | adrs/, patterns/ |
| cc-review | - | lessons/ |
| cc-retro | lesson, pattern | - |
| cc-kb | (整理发布) | All |
