# Req - 需求管理与规划

## 🎯 核心定位

**将用户模糊描述转化为可执行的结构化需求文档，主动挖掘而非被动记录**

> ⚠️ **边界**: 只管"要什么"，不管"怎么做"。不做任何架构/技术选型决策。

## 🔄 主流程（统一决策树）

```
用户输入需求
    ↓
[Phase 1] 原始记录 + 去重检查
    ↓
[Phase 2] 需求挖掘（主动发现 + 行业调研）
    ↓
[Phase 3] 规模判定 ← ──────────────────┐
    ↓                                  │
  简单？──是──→ [4A] 直接文档化        │
    ↓ 否                              │
  [4B] 复杂需求拆解                    │
    ├─ 模块划分                        │
    ├─ 接口契约（仅跨模块）            │
    └─ Feature 拆分                    │
    ↓                                  ↓
[Phase 5] 输出 + 优先级确认 ←──────────┘
```

---

## 📝 各阶段详解

### Phase 1: 输入与去重

**动作**:
- 记录原始描述到 `{project-path}/wiki/raw/{slug}.md`
- 检查已有需求文档，处理重叠：

| 重叠度 | 处理方式 | 自动建议 |
|--------|---------|---------|
| > 80% | 提示合并 | 展示差异点，一键合并 |
| 30-80% | 用户决定 | 预生成拆分/独立方案 |
| < 30% | 建议 | 默认独立，可手动关联 |

**输出**: `wiki/raw/{slug}.md` (纯文本，保留用户原话)

---

### Phase 2: 需求挖掘 ⭐ *这是 cc-req 的核心价值*

**原则**: 默认主动发现更多需求，整理完毕后一次性呈现给用户确认。

#### 2.1 显性需求提取

从用户描述快速提取功能点：

```
用户说："我要做一个登录系统"
↓ 提取
[1] ✅ 用户注册/登录（必选）
[2] ☐ 密码找回
[3] ☐ 第三方登录
...
```

#### 2.2 行业调研（动态深度）

**根据需求模糊度自动调整调研强度**:

| 场景 | 调研次数 | 示例 |
|------|---------|------|
| 清晰且常见 | 0-1 次（验证即可） | 登录功能、CRUD |
| 模糊或创新 | 2-3 次（充分探索） | AI推荐引擎、区块链存证 |
| 涉及合规/安全 | 必须调研行业标准 | 支付、医疗、金融 |

**调研内容**:
- 同类产品的功能清单和最佳实践
- 该类需求的常见场景和边界条件
- 安全性和合规性要求（如适用）

**规则**:
- 调研结果仅作为需求点候选，用户选择后才纳入
- 整理后一次性呈现清单，避免反复打扰

---

### Phase 3: 规模判定

**快速决策矩阵**（任一维度命中"复杂"列 → 进入拆解流程）:

| 维度 | 简单需求 | 复杂需求 (走拆解) |
|------|---------|-------------------|
| 模块数 | 1-2 | ≥ 3 |
| 文件改动 | < 10 | ≥ 10 |
| 迭代周期 | < 3 天 | ≥ 3 天 |
| 功能点 | 1-2 | ≥ 3 |
| 跨模块接口 | 无 | 有 |

**输出**: 判定结果 + 选择路径（4A 或 4B）

---

### Phase 4A: 简单需求文档化

**适用场景**: 单一功能、无需架构决策、可直接进入开发

**输出文件**: `{project-path}/wiki/road-map/{slug}.md`

```yaml
---
id: "{slug}"
type: requirement
status: approved          # draft → approved (支持渐进式)
title: "{标题}"
depends_on: []
created: "YYYY-MM-DD HH:MM"
updated: "YYYY-MM-DD HH:MM"
stale: false
priority: P0/P1/P2       # P0必须有 / P1应该有 / P2可以有
---

# {标题}

## 功能范围
{从 Phase 2 确认的需求点列表}

## 验收标准
{每个功能的完成定义}

## 优先级排序
{P0/P1/P2 分级及理由}
```

**✨ 中途退出机制**:
- 用户说"够了，先这样" → 输出当前版本，标记 `status: draft`
- 后续可通过 `cc-feat` 或再次调用 `cc-req` 迭代完善至 `approved`

---

### Phase 4B: 复杂需求拆解

**适用场景**: 多模块交互、涉及架构决策、需要先行设计

#### Step 1: 模块划分

**规则**: 按职责边界分（不是按文件/UI 分）

✅ 正确示例:
- 认证模块 / 支付模块 / 通知模块

❌ 错误示例:
- 登录页面 / 注册页面 / 忘记密码页面（这些是一个模块的 UI）

**每个模块输出卡片**:

```markdown
## 模块: {名称}
| 属性 | 值 |
|------|-----|
| 职责 | 一句话描述 |
| 边界 | 输入/输出/不做什么 |
| 依赖 | 依赖哪些其他模块 |
| 复杂度 | 接口数/数据表数/外部依赖数 |
| 风险点 | 已识别的技术风险 |

### 核心能力
- [ ] 能力 1
- [ ] 能力 2
```

#### Step 2: 接口契约草案

**只定义跨模块接口**（模块内部实现细节不在此阶段定义）:

```markdown
## 接口: {名称}
| 属性 | 值 |
|------|-----|
| 方向 | A → B (谁调谁) |
| 协议 | HTTP REST / gRPC / 内部函数 / Event |
| 格式 | JSON / Protobuf / TypeScript 类型 |

### Request / Response / Error Cases
{字段定义 + 错误码表}
```

**智能简化**:
- P0 接口：必须完整定义（请求/响应/错误码）
- P1/P2 接口：可简化为清单形式（方法名 + 参数列表），详细定义留待 arch 阶段

#### Step 3: Feature 拆分

**输出文件**: `{project-path}/wiki/road-map/{slug}.yaml`

```yaml
features:
  - id: "{feat-slug}"
    title: "标题"
    priority: p0              # p0(阻塞) / p1(重要) / p2(优化)
    depends_on: []            # 同需求内的其他 feat id
    scope_files: 5           # 涉及文件数（估算）
    modules: ["auth", "db"]   # 涉及模块
    status: todo              # todo | doing | done | skipped

shared_tasks:
  - id: setup-module
    title: "搭建骨架"
    status: todo
    blocks: ["feat-a", "feat-b"]
```

**拆分规则**:

| 规则 | 说明 |
|------|------|
| 单一职责 | 每个 feat 只做一个内聚功能点 |
| 可独立验收 | 完成后可单独测试上线 |
| 依赖最小化 | 减少 depends_on，允许并行开发 |
| P0 先行 | p0 无依赖或只依赖已完成 feat |

**质量门禁**:

| 检查项 | 通过标准 | 处理方式 |
|--------|---------|---------|
| 依赖图无环 | DAG 无循环依赖 | ❌ 必须修复 |
| 关键路径长度 | ≤ 5 层 | ⚠️ 建议优化 |
| P0 可并行度 | ≥ 2 个无依赖的 p0 feat | 💡 推荐调整 |

---

### Phase 5: 最终输出与确认

**产出物**:

| 文件 | 路径 | 适用场景 |
|------|------|---------|
| 需求文档 | `wiki/road-map/{slug}.md` | 所有需求 |
| Feature 清单 | `wiki/road-map/{slug}.yaml` | 仅复杂需求 |

**确认动作**:
- 呈现最终优先级分级：
  ```
  ✅ 已选需求点及优先级：

  P0（必须有）: 用户注册/登录、密码找回
  P1（应该有）: 第三方登录、Token 过期自动刷新
  P2（可以有）: 手机验证码登录、记住登录状态

  确认以上优先级？或调整后确认。
  ```

**✨ 渐进式完善**:
- 首次输出可为 `draft` 状态，包含已确认的内容
- 未尽事项标记为 TBD（To Be Determined）
- 后续迭代时只补充 TBD 部分，避免重复工作

## 🧠 AI 自由度框架

### L1: 严格约束（不可违反）

```yaml
必须遵守:
  - 输出路径规范: wiki/raw/, wiki/road-map/
  - 文件命名: kebab-case 语义化
  - 核心流程顺序: 去重 → 挖掘 → 判定 → 输出
  - 质量门禁: DAG无环、依赖合理
  
禁止行为:
  - 跳过规模判定直接输出
  - 忽略已有需求重复检查
  - 做技术选型或架构决策（这是 cc-arch 的职责）
```

### L2: 引导性建议（推荐但可偏离）

```yaml
推荐做法:
  - 调研时参考 2-3 个同类产品
  - 模块按职责边界划分
  - Feature 拆分遵循单一职责原则
  - 优先级采用 P0/P1/P2 三级分类

可偏离场景:
  - 用户明确要求特定的划分方式
  - 项目存在特殊架构约束
  - 时间紧迫需要快速输出 MVP 版本
  - 需求本身非常清晰且无需深挖
```

### L3: 完全自由（AI 自主判断）

```yaml
自主决策范围:
  - 需求描述的语言风格和详细程度
  - 清单呈现的顺序和分组方式
  - 是否主动补充行业最佳实践建议
  - 风险识别的详细程度（简单列举 vs 深度分析）
  - 里程碑划分的粒度（粗略阶段 vs 详细任务）
  
决策依据:
  - 项目类型（开源/企业/个人）
  - 团队经验（新手/资深）
  - 时间压力（宽松/紧迫）
  - 需求清晰度（明确/模糊）
```

**使用示例**:
```
用户输入: "做一个登录功能"

AI 可选择:
a) 快速模式 (时间紧/需求清):
   → 直接输出 3 个核心需求点（注册/登录/找回密码）
   → 跳过行业调研，标记 status: draft
   
b) 标准模式 (常规场景):
   → 挖掘后输出 8 个需求点（含安全/体验相关）
   → 1次调研验证，完整 frontmatter
   
c) 深度模式 (企业项目/安全敏感):
   → 调研后输出 15 个需求点（含合规/国际化/审计）
   → 完整的模块划分和接口契约草案
```

## 🔗 下游协作

| 当前状态 | 下一步技能 | 触发条件 | 交接内容 |
|---------|-----------|---------|---------|
| 简单需求 (`approved`) | `cc-feat` | 单一功能、无需架构决策 | 需求文档 + 验收标准 |
| 复杂需求拆解完 | `cc-arch` | 多模块、需先确认架构 | 模块划分 + 接口契约 + Feature 清单 |
| 架构设计完成 | `cc-feat` | 进入开发实现 | 架构文档 + 技术选型 |

**上下文交接协议**:
- 使用 frontmatter 的 `status` / `stale` 标记文档状态
- 下游技能读取时若发现 `stale: true`，必须先同步再继续
- 传递关键信息：决策结论、待解决问题、依赖假设

## 🔙 渐进细化机制

当下游技能发现需求不足时，可回退补充（不重走完整流程）：

| 触发场景 | 回退动作 | 补充内容 | 更新范围 |
|---------|---------|---------|---------|
| cc-arch 发现需求边界不清 | ↩ 补充边界问题 | 用例场景、约束条件 | REQ 文档 + stale 标记 |
| cc-feat 发现接口定义缺失 | ↩ 补充接口层需求 | API 契约、数据格式 | REQ 文档 + stale 标记 |
| cc-fix 修复 bug 发现需求遗漏 | ↩ 补充边界条件 | 异常场景、错误处理 | REQ 文档 + stale 标记 |

**规则**:
- 回退只补缺失点，不重新走完整流程
- 补充后更新 REQ 文档的 `updated` 时间戳
- 如变更重大，设 `stale: true` 并触发变更传播（详见 AGENTS.md）

## 🚫 常见问题处理

| 问题 | 处理方式 | 示例 |
|------|---------|------|
| 需求与现有 REQ 重叠 | 智能冲突解决（Phase 1） | "检测到 85% 重叠，建议合并为 user-auth-v2" |
| 需求涉及多个不相关功能 | 拆分为多个独立 REQ | "支付+通知→拆分为 payment-gateway + notification-service" |
| 用户描述模糊 | 深挖需求点（Phase 2） | "做一个后台"→澄清为"CMS/管理面板/数据分析平台" |
| 规模超标需拆解 | 自动进入复杂流程 | 3个模块→走 Phase 4B 拆解 |
| 用户中途改变需求 | 增量更新而非重做 | 新增需求点→追加到清单，重新确认优先级 |
| 时间紧迫要快速输出 | 采用 L3 自由度的快速模式 | 输出 draft 版本，后续迭代完善 |

## 📊 与原版本对比

| 维度 | 原版本 | 新版本 | 改进点 |
|------|--------|--------|--------|
| 流程结构 | 简单(1-4) + 复杂(1-6) 双轨制 | 统一 5 阶段流水线 | 消除重复，逻辑更清晰 |
| 调研策略 | 固定限 2 次搜索 | 动态深度（0-3次按需） | 适配不同需求复杂度 |
| 文档状态 | 一次性 approved | 支持 draft → approved 渐进式 | 符合敏捷开发理念 |
| AI 自由度 | 刚性规则为主 | L1/L2/L3 三级框架 | 平衡一致性与适应性 |
| 冲突处理 | 机械式提示 | 智能分析 + 一键操作建议 | 减少用户认知负担 |
| 代码行数 | ~289 行 | ~220 行（更聚焦） | 可读性提升，维护成本降低 |

**功能一致性保证**: ✅ 所有原有功能均已保留，仅优化结构和增加灵活性。
