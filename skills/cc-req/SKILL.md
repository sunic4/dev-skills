# Req - 需求管理与规划

## 职责

收集用户需求 → 分析规模 → 简单需求直接文档化 / 复杂需求拆解为子 feature 清单

**核心原则**: 只管"要什么"，不管"怎么做"。不做任何架构/技术选型决策。

## 触发条件

- 用户描述新功能/新需求/用户故事

---

## 规模判定（Step 0）

| 维度 | 简单需求 | 复杂需求 (走拆解) |
|------|---------|-------------------|
| 模块数 | 1-2 | **≥ 3** |
| 文件改动 | < 10 | **≥ 10** |
| 迭代周期 | < 3 天 | **≥ 3 天** |
| 功能点 | 1-2 | **≥ 3** |
| 跨模块接口 | 否 | **是** |

**任一维度命中复杂需求列 → 进入拆分流程**

---

## 简单需求流程 (Step 1-4)

### Step 1: 原始输入 + 去重

```
用户描述 → `raw/inputs/{slug}YYYYMMDDHHMMSS.md`
```

深挖前检查已有需求：完全相同→提示更新；部分重叠→用户决定合并/拆分/独立；无重叠→正常继续。

### Step 2: 需求发现 + 整理

**默认主动发现更多需求，整理完毕后一次性呈现给用户确认。**

1. 基于已有知识快速提取需求点
2. 网络调研补充同类产品/最佳实践中的常见需求点（自动执行，无需用户触发）
3. 合并去重后，按类别整理为完整清单

**网络调研规则**：
- 搜索同类产品的功能清单和最佳实践
- 搜索该类需求的常见场景和边界条件
- 调研结果仅作为需求点候选，用户选择后才纳入
- **限时**：调研不超过 2 次搜索，避免耗时过长

整理完毕后呈现清单：

```
📋 需求点清单（请多选）：

核心功能：
  [1] ✅ 用户注册/登录（必选）
  [2] ☐ 第三方登录（微信/Google）
  ...

请选择需要的需求点编号（可多选），或补充其他需求：
```

### Step 3: 优先级确认

用户选择完成后，直接根据选择结果确认优先级：

```
✅ 已选需求点及优先级：

P0（必须有）: 用户注册/登录、密码找回
P1（应该有）: 第三方登录、Token 过期自动刷新
P2（可以有）: 手机验证码登录、记住登录状态

确认以上优先级？或调整后确认。
```

### Step 4: 输出需求文档

输出到 `req/{slug}.md`

Frontmatter:
```yaml
---
id: "{slug}"
type: requirement
status: approved
title: "{标题}"
depends_on: []
created: "YYYY-MM-DD HH:MM"
updated: "YYYY-MM-DD HH:MM"
stale: false
---
```

正文结构（精简版，small 项目 3 章即可）：
1. 功能范围
2. 验收标准
3. 优先级

## 复杂需求拆解流程 (Step 1-6)

### Step 1: 原始输入 + 去重

同简单需求流程 Step 1。

### Step 2: 需求发现 + 整理

同简单需求流程 Step 2。

### Step 3: 模块划分

**规则**: 按职责边界分 (不是按文件/组件分)
- ✅ 认证模块 / 支付模块 / 通知模块
- ❌ 登录页面 / 注册页面 / 忘记密码页面 (这些是一个模块的UI)

每个模块输出卡片:

```markdown
## 模块: {名称}
| 属性 | 值 |
|------|-----|
| 职责 | 一句话描述 |
| 边界 | 输入/输出/不做什么 |
| 依赖 | 依赖哪些其他模块 |
| 复杂度 | high / medium / low |
| 风险点 | 已识别的技术风险 |

### 核心能力
- [ ] 能力 1
- [ ] 能力 2
```

### Step 4: 接口契约草案

只定义**跨模块**接口:

```markdown
## 接口: {名称}
| 属性 | 值 |
|------|-----|
| 方向 | A → B (谁调谁) |
| 协议 | HTTP REST / gRPC / 内部函数 / Event |
| 格式 | JSON / Protobuf / TypeScript 类型 |

### Request / Response / Error Cases
{字段定义 + 错误码表}
```

### Step 5: 拆子 Feature 清单

生成 `req/{slug}.yaml`:

```yaml
features:
  - id: "{feat-slug}"
    title: "标题"
    priority: p0              # p0(阻塞) / p1(重要) / p2(优化)
    depends_on: []            # 同需求内的其他 feat id
    estimated_files: 5        # 预估文件数
    modules: ["auth", "db"]   # 涉及模块
    status: todo              # todo | doing | done | skipped

shared_tasks:
  - id: setup-module
    title: "搭建骨架"
    status: todo
    blocks: ["feat-a", "feat-b"]
```

**拆分规则**:

| 规则 | 说明 |
|------|------|
| 单一职责 | 每个 feat 只做一个内聚功能点 |
| 可独立验收 | 完成后可单独测试上线 |
| 依赖最小化 | 减少 depends_on，允许并行 |
| P0 先行 | p0 无依赖或只依赖已完成 feat |

**质量检查**:

| 检查项 | 通过标准 |
|--------|---------|
| 依赖图无环 | DAG 无循环依赖 |
| 关键路径长度 | ≤ 5 层 |
| P0 可并行度 | ≥ 2 个无依赖的 p0 feat |

### Step 6: 输出结构

输出到 `req/{slug}.md` + `{slug}.yaml`

```markdown
---
id: "{slug}"
type: requirement
status: planning
title: "{标题}"
depends_on: []
created: "YYYY-MM-DD HH:MM"
updated: "YYYY-MM-DD HH:MM"
stale: false
---

# {标题}

## 背景

## 模块划分
{Step 3 产出}

## 接口契约
{Step 4 产出}

## 子 Feature 清单
见附件 `{slug}.yaml`

## 里程碑
| 阶段 | 目标 | 包含 feat | 预期产出 |
| Phase 1 | 核心跑通 | feat-a, feat-b | 可用但缺边界 |
| Phase 2 | 完善体验 | feat-c, feat-d | 生产可用 |

## 风险与缓解
| 风险 | 影响 | 缓解措施 |
```

## 状态流转

| 转换 | 触发 | 动作 |
|------|------|------|
| draft → approved | 简单需求质量门禁通过 | 直接标记，触发变更传播 |
| planning → approved | 复杂需求里程碑确认 | 标记 |
| approved → implemented | feat-accept 通过 | 更新 status |
| any → deprecated | 需求取消 | 标记下游全部 stale |

## 变更传播

变更传播核心机制详见 cc/SKILL.md "变更传播机制"。

REQ 特有传播规则:
1. 更新自身 `updated`，重大变更设 `stale: true`
2. Grep 找所有引用本 REQ 的 ARCH/FEAT 文档
3. 标记下游 `stale: true`

## 渐进细化

| 触发方 | 动作 |
|--------|------|
| cc-arch 发现需求边界不清 | ↩ 回退补充边界问题 |
| cc-feat 发现接口定义缺失 | ↩ 回退补充接口层需求 |
| cc-fix 修复 bug 发现需求遗漏 | ↩ 回退补充边界条件 |

回退只补缺失点，不重走完整流程。补充后更新 REQ + 触发变更传播。

---

## 与其他技能的协作

| 场景 | 下一步 | 条件 |
|------|--------|------|
| 简单需求 | → `feat` | 单一功能、无需架构决策 |
| 复杂需求 | → `arch` | 先确认模块划分和接口契约 |
| 需求涉及系统设计/技术选型 | → `arch` | 非 trivial、涉及多模块 |

---

## 输出规范

| 项目 | 格式 |
|------|------|
| 原始输入 | `raw/inputs/{slug}YYYYMMDDHHMMSS.md` |
| 需求文档 | `req/{slug}.md` (含 frontmatter) |
| 子feature清单 | `req/{slug}.yaml` (复杂需求时) |
| ID 命名 | kebab-case 语义化 |

## 常见问题

| 问题 | 处理方式 |
|------|---------|
| 需求与现有REO重叠 | 提示用户选择：合并/拆分/独立 |
| 需求涉及多个不相关功能 | 拆分为多个独立 REQ |
| 用户描述模糊 | 深挖需求点，整理后一次性呈现确认 |
| 规模超标需拆解 | 进入复杂需求流程 (Step 1-6) |
