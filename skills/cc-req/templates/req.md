# Req 输出文档结构

以下为 `{project-path}/wiki/road-map/{slug}.md` 的标准章节结构。

```markdown
---
id: "{slug}"
type: requirement
status: approved
title: "{标题}"
depends_on: []
created: "YYYY-MM-DD HH:MM"
updated: "YYYY-MM-DD HH:MM"
stale: false
---

# {标题}

## 背景
{为什么需要这个需求？解决了什么问题？}

## 用户故事
{按角色分组，可选}

### 作为 {角色}，我希望 {能力}，以便 {价值}

## 功能范围

### 核心功能
- [ ] 功能点 1
- [ ] 功能点 2

### 范围外 (Out of Scope)
- 明确不做的事项

## 约束
> 只描述现象和期望行为，不描述技术实现方案。

### 边界条件
- **数据**: {最大值/最小值/空值/超长} 用户应看到什么？
- **并发**: {多实例/同时操作} 用户应体验到什么？
- **异常**: {服务不可用/第三方失败} 用户应看到什么？
- **安全**: {数据可见性/操作鉴权} 策略？

### 非功能性约束
| 维度 | 要求 | 依据 |
|------|------|------|
| 性能 | {如 60fps @ 1000 节点} | {依据或 TBD} |
| 兼容性 | ... | ... |

## 验收标准
{可测试的、具体的、通过/不通过的判断依据}
- [ ] 标准 1
- [ ] 标准 2

## 优先级与排期
| 优先级 | 功能 | 备注 |
|--------|------|------|
| P0 | 必须有 | MVP 核心闭环 |
| P1 | 应该有 | 完善体验 |
| P2 | 可以有 | 锦上添花 |

## 关联文档
- 架构: {arch 文档路径, 如已创建}
- Roadmap: {road-map 路径, 如已创建}
```

**规则**:
- 每个章节可选（不适用则跳过），但 `功能范围` 和 `验收标准` 必须有
- **需求/方案分离**: 只描述"是什么"和"为什么"，不描述"怎么做"
  - 约束写现象（用户应看到/体验到什么），不写方案（用什么技术实现）
  - 技术约束（如包体积 < 50KB、必须支持 IE11）记录在约束章节，技术选型决策归 cc-arch
